**YAGNI (You Aren't Gonna Need It)** — это принцип разработки программного обеспечения, согласно которому **не стоит реализовывать функционал, который вам пока не нужен**.
## Основная идея

  chart="

  B --> C[Усложнение системы]
  C --> D[Больше багов]
  D --> E[Труднее поддерживать]

/>

- **Избегать избыточной функциональности**  
  Преждевременное добавление возможностей "на будущее" приводит к:
  - Усложнению архитектуры
  - Трате времени на ненужный код
  - Проблемам с поддержкой

- **Сфокусироваться на текущих задачах**  
  Лучше реализовывать функционал, когда:
  - Есть четкие требования
  - Появилась реальная необходимость
  - Можно сделать более осознанно

**Плохой подход (нарушение YAGNI):**
```javascript
function processData(array) {
  // Фильтруем
  let filtered = array.filter(item => item.active);

  // Сортируем по нескольким полям (хотя сейчас нужен только фильтр)
  filtered.sort((a, b) => a.priority - b.priority || a.id - b.id);

  // Храним дополнительную статистику (не требуется по ТЗ)
  const stats = {
    count: filtered.length,
    maxPriority: Math.max(...filtered.map(f => f.priority))
  };
  
  // Предполагаем, что в будущем пригодится лог для отладки
  console.log("DEBUG: processed data", filtered, stats);

  return filtered;
}
```

**Хороший подход (по YAGNI):**
```javascript
let filtered = array.filter(item => item.active);
```
## Когда применять YAGNI?

<Mermaid
  chart="

  Разработка --> Реализовать: Есть конкретная потребность

1. **На ранних этапах проекта**  
   Когда требования ещё не стабилизировались

2. **При прототипировании**  
   Когда важно быстро проверить идею

3. **Для неосновного функционала**  
   Когда вероятность использования низкая

### Вывод 
YAGNI помогает сохранять код простым и поддерживаемым, избегая ненужной сложности. Реализуйте функционал только тогда, когда он действительно понадобился, а не когда вам кажется, что он может понадобиться в будущем.
